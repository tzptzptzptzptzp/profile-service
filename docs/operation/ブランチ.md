# ブランチ

Webアプリケーション開発において、効率的で整理されたバージョン管理を実現するために、以下のブランチ戦略を採用します。

## 基本ブランチ

- **main**:
  - 本番環境
  - リリース可能な状態のコードを常に保持する

- **develop**:
  - 検証環境
  - 次のリリースのための開発コードを集約する
  - 新機能や修正はまずこのブランチに統合される

## ブランチの運用ルール

1. **新規ブランチの作成**:
   - 開発は、`develop`ブランチから新しいブランチを作成する
   - ブランチ名は、`task/` or `fix`プレフィックスを使用し、その後に明確なタスク内容を示す
   - 例: `task/#7_add_operation_document`, `fix/#12_fetch_user_data`

2. **ブランチの命名規則**:
   - `task/<issue番号>_<内容>`
   - 具体例: `task/#7_add_operation_document`

3. **作業の統合**:
   - タスクが完了したら、ブランチを`develop`にプルリクエストでマージする
   - マージ時にはレビュープロセスを経て、コードレビューやテストを行う

4. **ブランチの削除**:
   - マージが完了したタスクブランチは不要なものとして削除する

## 注意事項

- **コンフリクトの解消**:
  - `develop`ブランチとのコンフリクトが発生した場合、速やかに解消する

- **リリース**:
  - `develop`ブランチが安定し、すべての機能が完成次第、`main`にマージする

この戦略を使用して、効率的かつ整理された開発プロセスを維持し、プロジェクトの成功につなげます。
