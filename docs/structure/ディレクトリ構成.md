# ディレクトリ構成

Webアプリケーション開発において、効率的で整理されたプロジェクト構成を実現するために、以下のディレクトリ戦略を採用します。

## 各ディレクトリの説明

### root/

プロジェクトのルートディレクトリです。プロジェクト全体に関わる設定ファイルやソースコードの主要ディレクトリを含みます。

- **.github**: GitHubに関連する設定やワークフローを配置する
- **.vscode**: VS Codeの設定ファイルを配置する
- **docs**: プロジェクトに関するドキュメントを管理する
- **src**: ソースコードの主要ディレクトリ
- **public**: 公開されるリソース（画像、アイコンなど）を配置する
  - **images**: プロジェクトで使用する画像ファイルを格納する（`jpg, png, gif`）
  - **icons**: プロジェクトで使用するアイコンファイルを格納する（`svg`）

### docs/

プロジェクトの設計や仕様に関するドキュメントを格納します。

- **image**: ドキュメントで使用する画像を格納する
- **operation**: プロジェクト運用に関するドキュメントを格納する
- **requirement**: 要件定義書を格納する
- **screen**: 画面仕様やモックアップを格納する
- **specification**: 各機能の仕様書を格納する
- **structure**: システム構造に関するドキュメントを格納する
- **technology**: 技術仕様や選定技術に関するドキュメントを格納する
- **testing**: テスト仕様や計画を格納する
- **architecture**: アーキテクチャに関する情報を格納する
  - **api**: API設計に関するドキュメントを格納する
  - **db**: データベース設計に関するドキュメントを格納する

### src/

アプリケーションのソースコードを格納します。

- **app**: アプリケーションの主要なコードを配置する
- **components**: 再利用可能なUIコンポーネントを配置する
- **configs**: 設定ファイルを管理する（例: `[name].config.ts`）
- **features**: 特定の機能に関連するグループを配置する
- **hooks**: カスタムフックを配置する（例: `[name].hook.ts`）
- **providers**: コンテキストプロバイダを配置する（例: `[Name].provider.tsx`）
- **schemas**: データ構造やバリデーションのスキーマを配置する（例: `[name].schema.ts`）
- **stores**: 状態管理関連のファイルを配置する（例: `[name].store.ts`）
- **styles**: スタイルシートを配置する（例: `[name].style.scss`）
- **types**: TypeScriptの型定義を配置する（例: `[name].type.ts`）
- **utils**: ユーティリティ関数を配置する（例: `[name].util.ts`）

### app/

個別の機能やページに関連するコードを格納します。

- **api**: Next.jsのAPIルートに関連するコードを配置する（例: `[name].api.ts`）
- **[name]**: 個別のページを管理する
  - **index.tsx**: ページのエントリーポイント
  - **layout.tsx**: ページレイアウトを定義
  - **page.tsx**: ページのコンテンツを定義

### components/

再利用可能なUIコンポーネントを目的別に配置します。

- **elements**: 基本的なUIコンポーネントを配置する（例: `[Name].element.tsx`）
- **forms**: フォーム関連コンポーネントを配置する（例: `[Name].form.tsx`）
- **layouts**: ページ全体のレイアウトを配置する（例: `[Name].layout.tsx`）

### features/

特定の機能に関連するコードをグループ化して管理します。

- **[name]**: 機能ごとに関連するコードをまとめる
  - **components**: 機能に特化したコンポーネントを配置する（例: `[Name].component.tsx`）
  - **configs**: 機能に関連する設定を配置する（例: `[name].config.ts`）
  - **hooks**: 機能に関連するカスタムフックを配置する（例: `[name].hook.ts`）
  - **schemas**: 機能に関連するスキーマを配置する（例: `[name].schema.ts`）
  - **styles**: 機能に関連するスタイルを配置する（例: `[name].style.scss`）
  - **types**: 機能に関連する型定義を配置する（例: `[name].type.ts`）
  - **utils**: 機能に関連するユーティリティ関数を配置する（例: `[name].util.ts`）

## tsxの設計ルール

- `page.tsx`や`layout.tsx`では`use client`を使用せず、**Server Component**として設計する  
これは、サーバーサイドレンダリング（SSR）を活用することで、パフォーマンスとSEOの向上を目指すためである
  
- `page.tsx`から呼び出せるUIコンポーネントは、同階層に位置する`index.tsx`のみとする  
これにより、コンポーネントの依存関係が適切に管理され、コードの可読性とメンテナンス性が向上する

- `page.tsx`でサーバーサイドで取得したデータを`index.tsx`に引き渡してコンテンツを作成する

- コンポーネントは可能な限りステートレスに保ち、ステート管理が必要な場合は、最小限のClient Componentで実装する

この設計ルールに従うことで、効率的な開発と高品質なアプリケーションの実現を目指します。

## ディレクトリ構造図

```
root/
 ├── .github
 ├── .vscode
 ├── docs
 ├── src
 ├── public
     ├── images
     ├── icons

docs/
 ├── image
 ├── operation
 ├── requirement
 ├── screen
 ├── specification
 ├── structure
 ├── technology
 ├── testing
 ├── architecture
      ├── api
      ├── db

src/
 ├── app
 ├── components
 ├── configs / [name].config.ts
 ├── features
 ├── hooks / [name].hook.ts
 ├── providers / [Name].provider.tsx
 ├── schemas / [name].schema.ts
 ├── stores / [name].store.ts
 ├── styles / [name].style.scss
 ├── types / [name].type.ts
 ├── utils / [name].util.ts

app/
 ├── api / [name].api.ts
 ├── [name] / index.tsx
      ├── index.tsx
      ├── layout.tsx
      ├── page.tsx
 
components/
 ├── elements / [Name].element.tsx
 ├── forms / [Name].form.tsx
 ├── layouts / [Name].layout.tsx
 
features/
 ├── [name]
      ├── components / [Name].component.tsx
      ├── configs / [name].config.ts
      ├── hooks / [name].hook.ts
      ├── schemas / [name].schema.ts
      ├── styles / [name].style.scss
      ├── types / [name].type.ts
      ├── utils / [name].util.ts
```

このディレクトリ構成により、プロジェクトのコードは適切に整理され、各部分の役割が明確になります。
